每條邊：觸發者 / 守衛 / 後續動作 / 例外

合法邊白名單 VALID_TRANSITIONS 的單一權威在 TS 端；所有寫入皆走 RPC，在 DB 端單一交易內完成 CAS UPDATE orders ＋ INSERT order_status_log，任一段失敗整段 rollback（T110）。

E1pending_payment → paid
  • 觸發ECPay webhook（權威）／ reconcile cron（兜底）— ensureOrderPaid
  • 守衛條件式 CAS WHERE status='pending_payment'，只有搶到的請求推進
  • 後續動作清購物車（T75, best-effort）· 寄確認信 · 通知店家 · 開電子發票
  • 例外CAS 沒搶到 → 複查：在 lineage＝already-settled；非 lineage＝closed（P0 告警）；查不到＝indeterminate
B1pending_payment → cancelled
  • 觸發逾期 cron（T66, 72h）／ 後台取消 — transitionOrder
  • 守衛canTransition ✅ 且無 paid payment（findPaidPayment）— 取消守衛下沉於此，所有取消路徑必經
  • 後續動作寫稽核 log；TOCTOU 補查（見 §3）
  • 例外有 paid → PaidOrderCancelBlockedError；cron 記 paidConflict＋告警、結帳回錯不建新單、admin 導退款
E2paid → in_production
  • 觸發後台手動 — transitionOrder
  • 守衛canTransition ✅ · CAS WHERE status='paid'
  • 後續動作寫稽核 log（含 actorId）
  • 例外被別流程搶動 → OrderTransitionRaceError（良性競態，跳過非失敗）

  • 觸發後台手動 — transitionOrder
  • 守衛canTransition ✅ · CAS
  • 後續動作寫 tracking_no · 寄出貨通知信
  • 例外狀態已轉但單號寫入失敗 → 回 warning（非 error），提示用「修正物流單號」補填
E4shipped → completed
  • 觸發後台手動 — transitionOrder
  • 守衛canTransition ✅ · CAS
  • 後續動作寫稽核 log
  • 例外競態 → OrderTransitionRaceError

  • 觸發後台退款區塊（T47）— refundOrder → refund_order RPC

  • 後續動作payment refunded · order refunded · 稽核 log · 退款通知信（sendOnce 去重）
  • 例外仍有 paid 殘留 → RefundPaymentNotFlippedError；pending／cancelled 不可走退款區塊（server 端擋）

逃生口 · adminOverrideStatus

繞過 VALID_TRANSITIONS，可改任意狀態（operatorId ＋ reason 必填）；仍走同一 RPC，享 CAS 守衛（防雙擊／兩位管理者送互斥目標，只一筆成功）。前置擋 to === from——目標＝現況時 SET 不改動 WHERE 欄位，READ COMMITTED 下 CAS 會因 EvalPlanQual 失效。

不翻 payment、不寄通知信——與正規退款的關鍵差異。留下的半套狀態（訂單 refunded / payment 仍 paid）由 reconcile paid payment on refunded order durable 稽核臂每日告警，直到人工「補登記退款」（repair_refunded_payment RPC, 0021）。

03

並發時序：取消守衛的 TOCTOU 窄窗

最危險的競態——逾期 cron 正要取消訂單的同時，webhook 才把 payment 翻 paid。pre-guard 查完之後、CAS 取消 commit 之前的毫秒窄窗內若付款成立，主對帳臂（鍵 payment=pending）與漂移臂（鍵 orders=pending_payment）都撈不到——commit 後再查一次即偵測。

durable 兜底：即使 t4 這次查詢失敗（只降 warning），reconcile 每日 recurring 稽核臂（鍵 payment=paid ∧ orders=cancelled）仍會每日重新撈到並告警，直到人工處理——不依賴取消當下的單次訊號。

另一個競態 · CAS 雙寫防護

兩個近乎同時抵達的 webhook 重送同時要把 pending_payment → paid：條件式 CAS WHERE status = p_from 保證只有一個搶到、只寫一筆 order_status_log（該表無 unique 約束）。沒搶到者複查狀態、回 already-settled 安靜返回，不重複記帳、不重複寄信。

04

金流兜底：對帳三臂

webhook 是即時路徑，reconcile cron 是最終防線。三條臂各有一組不相交的候選鍵——以某一鍵查詢的臂看不見另一鍵的漂移單，所以必須三臂並存，才覆蓋「錢已收但訂單未反映」的所有狀態。

06

QA 與情境

把 ensure-paid.ts 在「CAS 沒搶到 payment 推進」後的三態複查（already-settled / closed / indeterminate）與其真實情境展開——這是理解本狀態機在並發/重送/兜底下如何保持冪等又不誤報的關鍵。

already-settled
CAS 沒搶到 ∧ 現況 ∈ PAID_LINEAGE
付款其實早就成立過了，這次是重複進來、沒事可做。
安靜返回，不計 promoted、不重複寄信/開票/記帳。
closed
CAS 沒搶到 ∧ 現況 = cancelled/refunded
訂單已進終態，卻確認收到錢——錢與訂單狀態矛盾。
P0 告警 → 人工裁決（退款或恢復訂單，ops-runbook §6.1）。
indeterminate
CAS 沒搶到 ∧ 複查失敗/查無此單
連訂單現況都查不到，無法確認。
P0 告警＋保守不動作，留給下次重試判定。
already-settled 是什麼情況？

CAS 用條件式 UPDATE ... WHERE status='pending_payment'，搶不到（0 rows）代表訂單已被別人搶先推進。複查發現它在 PAID_LINEAGE 上，就是這態。「別人」是誰：

  • 綠界 webhook 重送：第一次已翻 paid/寄信/開票，第二次重送搶不到——冪等的核心，重送不重複扣帳/寄信。
  • webhook 與對帳併發：webhook 先推進，稍後對帳主臂也來，CAS 搶不到（不重複計 promoted）。
  • 對帳隔日重跑：前一天已推進，今天再撈到。
  • admin 已推進到製作/出貨：付款成立後你把訂單推到 in_production，遲到的重送才進來。

為什麼判斷用整條 PAID_LINEAGE 而非只認 paid？看最後一種：若只認 paid，訂單已被推到 in_production 就會被誤判成「付款卡住」發 P0——但那是正常演進。用「paid 及其一切後續」判斷，才不會把正常狀態推進誤報成錢務異常。

TL;DR　「不是我推進的，但它已經是（或越過）paid 了，一切正常，我什麼都不用做。」
closed（錢收在已關閉訂單）的實際情境？

不是 bug，是取消/退款與付款的時間差競賽。最典型：

  • 逾期取消 vs 遲到付款：72h 逾期取消 cron 執行時 payment 還 pending → 取消放行 → 客人卻在快到期時才付、款項在取消後才於綠界完成 → 錢收了、訂單已 cancelled。
  • 人工取消後客人才付：你在後台取消 pending 單，客人拿還開著的舊連結付了款。
  • 取消守衛 TOCTOU 毫秒窄窗：pre-guard 查的瞬間還沒 paid → 放行取消 → commit 後 webhook 才翻 paid。
  • 退款後兄弟交易才成立：客人刷兩次，你依已付那筆退款結案，綠界之後才回報第二筆也成功 → 掛在已退款單上（客人被扣兩次、你只退一筆）。
  • Admin Override 逃生口：直接把訂單改 refunded 但不翻 payment → 半套狀態。

偵測：對帳主臂/漂移臂 → money received on closed order；取消當下複查 → money received on order cancelled during transition；每日 durable 稽核臂 → paid payment on cancelled/refunded order（每日重複告警直到處理）。

為什麼 payment 照翻/維持 paid？gateway_trade_no／raw_callback 是日後退款的唯一依據，錢的事實必須落地。

TL;DR　唯一「錢收了卻可能對不上」的路徑，永不自動修、每日告警催辦，人工裁決退款或恢復訂單。
indeterminate 什麼情況會發生？

CAS 沒搶到後的複查「查不到現況」，兩個子情況：

  • 複查 SELECT 暫時失敗（statusError）：Postgres timeout、連線池耗盡、與 Supabase 的網路抖動——Supabase 回 {error} 不 throw。
  • 查無此單（!current）：orderId 對不到任何列（正常流程近乎不可能，帳務表禁硬刪，屬防禦性處理）。

為什麼跟 closed 嚴格分開？closed＝讀到了、確認是 cancelled/refunded；indeterminate＝根本沒讀到。不能憑「我沒查到」就啟動退款/恢復訂單的錢務裁決——那是基於未知資訊對錢做決定。

TL;DR　「我這輪確認不了」，不是「確認有問題」——只告警＋保守不動，判定留給重試。
indeterminate 之後，重查就會出現真狀態嗎？

絕大多數會——這正是設計的自癒。暫時性故障會恢復，下次重試 DB 正常就讀到真狀態。但兩個前提：

  • 重試真的會發生：webhook 重送額度有限，對帳 cron 才是 durable 保證——只要 payment 還 pending/訂單漂移，下輪一定再撈到。
  • 例外：查無此單那種不會變好：它的真狀態就是「不存在」，重查一直回查無，是異常本身、要人工查。
TL;DR　暫時性抖動→重查會好（cron 保證重查會再發生）；查無此單→重試不會好、要查。
「連續多輪同一筆 indeterminate」＝真問題，實際是什麼？

不是抖動、是結構性壞掉。先用「單筆 vs 多筆」分根因（看 Sentry extra.status 是「查詢失敗」還是「查無此訂單」）：

  • 只有某一筆持續（多半 !current）：orderId 指向不存在的訂單。成因＝識別碼互轉算錯（merchant_trade_no↔order_no 重組 slice 錯、複本失同步，本專案 T67 踩過的坑）或孤兒 payment。重試不會好，要查為什麼 payment 指向不存在的訂單。
  • 很多筆同時持續（多半 statusError）：DB 層結構性問題——Supabase 長事故、連線池耗盡成常態、或 migration 在 orders 加了約束/RLS/trigger 讓查詢持續失敗（回歸）。要查 DB 健康與最近的 migration。

⚠️ 背後常是客人已付款卻卡 pending、確認信沒寄——不是可放著的噪音，要優先查根因。

判準　偶發一次＝正常抖動放著會好；連續多輪同一筆＝真問題要查（單筆查 code/資料、多筆查 DB/migration）。
名詞：「兜底」與「後續動作」
  • 兜底：安全網／最後防線——「A 失敗時 B 接住補救」，層層往下接確保沒東西無聲掉落。例：金流兜底＝webhook 漏接時由每日對帳接住。
  • 後續動作（原稱副作用）：相對於「狀態轉換」這個主作用，因轉換而連帶觸發的其他動作（清車/寄信/開票/寫 log）。它們可獨立失敗、各有冪等與兜底，失敗不得回滾狀態轉換（開票失敗不擋金流）。
  • PAID_LINEAGE：付款成立契約集合＝paid / in_production / shipped / completed；決定哪些狀態視為「付款已成立」（補寄確認信、already-settled 判斷都用它）。
