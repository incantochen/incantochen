系統流程與 User Flow 統整

從「使用者看到什麼」一路往下拆到「壞掉時誰來救」——四層對齊視圖：使用者流程 → 系統流程（成立條件/例外）→ 系統錯誤自動處置 → 人工救援。本頁為 system-flow-and-user-flow.md 的儀表板視圖，內容以程式碼與 ops-runbook 接地。

每個節點的成立條件與例外處置——含跨流程訂單狀態機。

防禦性規則＋金流失敗自癒矩陣——暫時性故障自癒。

核心哲學：越往上越貼近人（體驗），越往下越貼近錢（救援）；暫時性故障在 L3 自癒，錢務問題落到 L4 人工裁決、絕不自動壓掉。

Level 1

User Flow & Experience

四條主動線，體驗原則：自助、價格透明、低摩擦、不推銷（呼應高端客群主導性、反感推銷）。

F1首購下單

目錄 → 詳情頁（配戴情境圖）→ 頁內配置器（寶石色→金屬色→規格→數量）→ 即時報價 → 購物袋 → 結帳 → 同意客製條款 → 綠界付款 → 結果頁

結帳即會員（免先註冊）；付款結果頁「數秒內給明確結果」，絕不讓客人看到「待付款」而驚慌
F2回訪查單

輸入 Email → 收信（OTP 驗證碼為主＋magic link 為輔）→ 輸碼登入 → 會員中心 → 訂單狀態時間軸／物流單號

OTP 讓 session 精準落在當下裝置（手機輸入、桌機收信也不卡）；magic link 落地頁「再按一次才消耗 token」防掃描器點掉
F3售後申請

訂單詳情 →「商品問題回報」→ 填說明＋同意告知 → 店家後台審核 → 狀態同步回客人端

半客製＝法定客製品、無七天鑑賞退；客戶端只有單一入口，退款走人工確認
F4全客製預約

首頁 custom 入口 → 說明頁 → 需求表單（品項/預算/想法/聯絡）→ 通知店家 → 人工一對一

MVP 僅捕捉需求，不接金流、不建訂單（完整報價鎖價＝Phase 3）
Level 2

系統流程（成立條件與例外）

以 F1 首購下單（核心閉環）為主。正常路徑成立條件：白名單選項合法 → 前端價＝後端重算價 → 勾選客製同意 → 綠界回報成功。

節點系統動作成立條件例外處置
配置器三層白名單（類別 applies_to→款式 ProductOption→值 ProductOptionValue）選值在白名單內前端不得繞過；非法值後端拒絕
加入購物袋寫 unit_price_snapshot＋config_snapshot 快照cart.guest_token＝httpOnly cookie（擁有權檢查）token 不符→拒絕（防亂猜 id 動別人車）
結帳驗價🔴 verify-prices 依 DB 白名單重算，絕不信前端前端價＝後端價不一致→更新快照＋回 priceUpdated、不建單（R/S/Q loop）
Email 辨識以 Email 找/建會員全新 email→自動建會員命中既有會員（T71）→不靜默掛單，導 /login OTP 驗證
建立訂單create_order_with_items RPC（待付款）＋同意時間戳驗價通過—
付款判定背景 webhook 為權威；驗 CheckMacValue、冪等去重、條件式 UPDATEwebhook 回報成功前端 redirect 早於 webhook→結果頁輪詢
三態對帳成功頁未確認→主動呼叫綠界訂單查詢 API綠界明確回成功/失敗「尚未確認」≠失敗：不可顯示「失敗請重試」（會雙重扣款）
付款成功訂單→已付款、寄確認信＋通知店家＋開電子發票—serverless 一律 await（禁 fire-and-forget）
逾期未付T66：pending 逾期（72h）自動取消；重試換新 trade no—有 paid payment 就不准取消（取消守衛）

F2 / F3 / F4 成立與例外

  • F2 查單：OTP 正確且未過期（碼長不固定，雲端 8 碼、本機 6 碼，採 4–10 位彈性）→ 建 session；碼錯/過期→友善錯誤＋一鍵重寄。magic link 落地頁須「再按一次」才消耗 token（防掃描器預取）。
  • F3 售後：訂單狀態可申請（非待付款/取消/退款）→ service role 重驗擁有權 → 建 support_request → 通知店家；重複申請不硬擋（人工處理，按鈕 disabled 防連點）。
  • F4 預約：欄位完整→建詢問紀錄→通知店家；不接金流、不建訂單。
  • 跨流程狀態機：待付款→已付款/處理中→製作中→已出貨→已完成；分支 已取消/已退款。每次變更寫 order_status_log，狀態推進與稽核 log 交易化（RPC，log 寫不進整筆 rollback）。詳見附錄。
Level 3

System Error and Handling

核心哲學：暫時性故障自癒、錢務問題永不自動修。全鏈路靜默失敗點皆接 Sentry。

防禦性程式規則（動對應模組必守）
  • SDK 錯誤必檢查：Supabase/Resend 用回傳值帶錯誤，每次解構檢查 error；「查詢失敗」≠「查無資料」。
  • 並發去重用條件式 UPDATE，且 SET 必須改動 WHERE 用到的欄位（否則 EvalPlanQual 兩邊都搶到）。
  • numeric 比對前先 Number()；serverless 禁 fire-and-forget（一律 await/after）；識別碼互轉單一出處；客人輸入插 HTML 前先 escape。
金流失敗自癒矩陣（webhook 已失靈為前提，客人已扣款）
失敗點客人體驗自動處置自癒時間
① 訂單推進失敗（DB 暫時錯）付了款仍顯示待付款Sentry＋cron unexpected+1隔日對帳重試
② payment 翻 paid 失敗幾乎無感（訂單已 paid、信照寄）隔日 CAS 補翻（短暫漂移屬正常）隔日
③ 確認信寄送失敗（T88）信晚到webhook 回 0 觸發重送 reclaim＋每日 sweep 補寄最慢隔天
④ webhook 側卡單（paid/pending）付了款仍待付款T127 漂移臂隔日冪等推進＋補確認信隔日
⑤⑥ 錢收在已取消/退款訂單訂單已取消卻被扣款durable 稽核臂每日告警需人工

兜底層次：Webhook（權威）→ 成功頁主動對帳查詢（即時）→ 每日 reconcile cron（三臂）→ email-pending 最後安全網。

金額不符（P0）

  • webhook 端 TradeAmt ≠ 系統金額 → 拒絕處理、訂單停 pending_payment、不記帳。理論上不該發生（伺服器驗價），出現即當日開 bug 調查。
webhook 失敗機率量級（工程推估，非實測）
層單次失敗率綠界重送能救最終漏接
網路暫時故障~0.01–0.1%✅ 幾乎必救接近 0
Vercel 平台事故≈99.95%+ 可用⚠️ 看事故長度~0.01–0.05%
應用層（DB 暫時錯/bug）~0.01–0.1%/請求✅ 回 ERR 觸發重送接近 0
設定錯誤（ReturnURL/金鑰）每次動設定 5–20% 人為出錯❌ 全滅型、不自癒一筆真實交易 100% 驗出

結論：設定正確前提下單筆「webhook 徹底失靈、要靠對帳搶救」約 0.01–0.1%（幾個月到幾年一次）。風險大頭在設定層，集中在 T35 換正式網域＋金鑰時——唯一預防是改完立刻打一筆真實小額交易驗 webhook 到達。

Level 4

Operation Handling（人工救援）

出處 ops-runbook.md。紀律：修復 SQL 前先跑診斷；能走 /admin/orders（寫稽核 log）就不裸改 SQL；帳務表永禁 DELETE。

權威修復順序

  • 綠界後台 ＞ payment ＞ orders ＞ 通知信——永遠從權威端往下游同步、不反向。
  • 先 payment、再 orders、最後補通知；一次一單（帶 order_no/merchant_trade_no ＋狀態守衛）。
情境判斷處置
訂單卡待付款、客人聲稱已付綠界後台以 merchant_trade_no 查權威狀態①等對帳兜底 ②手動觸發 cron ③仍不行才 SQL 改 payment＋Admin Override 推訂單
P0 金額不符告警比對 orders/綠界/raw_callback不可標 paid；多收退刷、少收請補刷；開 bug
缺 gateway_trade_no綠界查 TradeNoUPDATE 回填
通知信卡 failed（T88 已自動重試）連續 2–3 天同筆＝永久性Resend 查退信→聯絡客人→補寄→標 sent
疑似重複扣款兩筆都在綠界顯示已收款保留最早成功那筆，其餘綠界退刷、系統標 refunded
退款（T47 記錄式）售後確認需退款綠界退刷 → 退款區塊登記（refund_order RPC 原子）
錢收在已取消/退款訂單每日 durable 稽核臂告警先聯絡客人：退款 or Admin Override 恢復訂單
對帳 Cron 失敗Vercel Cron 紀錄＋pending 筆數是否累積查 cron 啟用/route 200；ECPay 403 限流有退避隔日重試

何時聯絡綠界（02-2655-1775）：綠界查無交易但客人有扣款證明、退刷失敗、查詢 API 持續 403。準備 MerchantID、MerchantTradeNo（19 碼）、綠界 TradeNo、交易日期與金額。

附錄

訂單狀態機與對帳三臂

這兩塊有完整的視覺化專頁（含每條邊守衛/後續動作、TOCTOU 時序、三態複查 QA）——此處僅摘要。

A · 訂單狀態機
  • 7 狀態：pending_payment → paid → in_production → shipped → completed；分支 cancelled / refunded（僅這兩個是終止態）。
  • PAID_LINEAGE＝paid 及其後續（決定哪些狀態視為付款已成立、該補寄確認信）。
  • 所有寫入走 transition_order_status RPC：CAS UPDATE＋order_status_log INSERT 同一交易，任一失敗整段 rollback（T110）。
  • 守衛：取消守衛（有 paid 就不准取消）、退款守衛（有 paid 殘留不准轉 refunded）、TOCTOU 補查。
B · 金流兜底對帳三臂（每日台北 02:00）
臂候選鍵動作處置
主臂payment=pending打綠界確認→推進訂單→翻 payment→補通知自癒
漂移臂paid ∧ orders=pending不打綠界，信任財務事實冪等推進＋清 sibling自癒
稽核臂paid ∧ orders∈{cancelled,refunded}只偵測，durable 每日告警人工裁決

另掛 failed-notification sweep（補寄失敗信）與 uninvoiced sweep（補開發票），皆冪等、限流 break 也照跑；任一子臂查詢失敗整支回 HTTP 500（fail-visible）。
