# 05-4. Production 上線檢查清單與 Regression Test 規劃

## 1. 上線檢查清單（Go-Live Checklist）

### 🔴 必過五項（任一未過即不得上線）

- [ ] **1. 所有 URL 已切正式**：全部端點去除 `-stage`；站內付 2.0 **雙網域都要切**（ecpg＋ecpayment，只切一個是最常見漏洞）；對帳走 vendor 正式網域。未切者所有交易皆為模擬、無真實金流。
- [ ] **2. 正式金鑰已生效**：MerchantID/HashKey/HashIV 換為正式帳號值（廠商後台取得）。未換者所有驗章永遠失敗。
- [ ] **3. 金鑰不在前端與版本控制**：以環境變數／Secret Manager 管理；檢查 git 歷史。
- [ ] **4. 回呼端點正式可達**：ReturnURL 等指向正式 HTTPS（443），外網可 POST；未達者訂單永遠停在待付款。
- [ ] **5. RtnCode 型別判斷正確**：CMV 類字串 `'1'`、AES-JSON 類整數 `1`。型別錯誤＝所有成功交易被判失敗。

### 🟡 帳號與環境

- [ ] 正式帳號申請並審核通過；環境切換由單一開關（`ECPAY_ENV` 類）控制，具快速回切能力
- [ ] 主機 NTP 校時（MerchantTradeDate UTC+8；查詢 TimeStamp 3 分鐘時效；AES Timestamp 10 分鐘時效）
- [ ] 對帳檔下載 IP 白名單已在廠商後台設定

### 🟡 安全

- [ ] TLS 1.2；回呼 URL 為 FQDN、80/443、非 CDN 後方、非中文網址
- [ ] 回呼來源 IP 白名單（向客服索取；**官方未公開**）
- [ ] CheckMacValue 驗證 timing-safe；AES Base64 標準 alphabet
- [ ] ItemName/TradeDesc 消毒（HTML 標籤/控制字元/WAF 關鍵字）；MerchantTradeNo ≤20 英數；金額正整數
- [ ] 回呼參數白名單；錯誤訊息不洩漏內部資訊；顯示回傳值前 HTML escape
- [ ] 金鑰輪換 SOP 與洩漏處置 SOP 已演練（見 `03-architecture/04-security.md` §3）

### 🟡 回呼與冪等

- [ ] ReturnURL／OrderResultURL／ClientBackURL 三者互異
- [ ] `1|OK` 精確格式＋HTTP 200；10 秒內回應（P95 < 3 秒）；耗時工作已佇列化
- [ ] PaymentInfoURL（用 ATM/CVS/BARCODE 時）／PeriodReturnURL（用定期定額時）已設定並處理取號成功碼（'2'／'10100073'）
- [ ] 冪等：條件式 UPDATE；SimulatePaid=1 不出貨；例外全捕獲

### 🟡 功能驗收（正式環境實測）

- [ ] 正式帳號完成至少一筆**小額信用卡真實交易**（含 3D）
- [ ] 主要付款方式逐一走通；ATM 取號資訊正確呈現
- [ ] **退款實測**（測試環境測不到）：小額訂單全額退款成功；避開 20:15–20:30
- [ ] 定期定額（如使用）：真實合約建立→確認首期→Cancel
- [ ] 發票（如使用）：真實開立→查詢→（必要時）作廢；字軌已設定且餘量足夠
- [ ] 對帳媒體檔含當日真實交易且金額一致；撥款對帳檔隔日 14:00 後可查
- [ ] 爭議款流程知情：無 API，綠界以 Email 通知；憑證保存 ≥180 天

### 🟡 監控與復原

- [ ] 監控上線：交易成功率、回呼接收率、驗章失敗率、回呼 P95、403 次數、對帳差異（門檻見 `04-flows/08` §4）
- [ ] Feature Flag 可即時暫停收款；降級策略表已演練（`04-flows/08` §5）
- [ ] 上線首日觀察：建單:回呼 ≈1:1、首筆真實交易成功、對帳差異=0、取號成功碼未被誤判

### 漸進式上線策略

1. 小額真實交易驗證 → 2. 先只開信用卡 → 3. 穩定後逐步開 ATM/CVS 等 → 4. 金流穩定後才串發票自動開立。

## 2. Regression Test 規劃

### 2.1 觸發時機與範圍

| 觸發 | 回歸範圍 |
|------|---------|
| 每次 commit | Unit 全量（簽章/加密/參數/狀態機/代碼字典，秒級） |
| 每次 PR | Unit＋Integration 全量（假綠界） |
| 每日排程 | E2E 關鍵路徑最小集（sandbox） |
| Release 前 | `02-test-cases.md` 全案例（U/I/E 層） |
| 綠界官方「更新歷程」頁異動 | 受影響 API 的 E 層案例＋能力矩陣文件更新 |
| 金鑰輪換後 | 簽章/加密 E2E 冒煙＋一筆小額真實交易 |
| 事故修復後 | 該事故的再現案例**必須**先寫入案例庫再修復（見 2.3） |

### 2.2 回歸核心清單（每次 release 必跑的不變量）

1. CheckMacValue／AES 官方測試向量全數通過（加密實作凍結性）。
2. 兩套 URL encode 互換必失敗的防呆案例。
3. 取號成功碼（'2'／'10100073'）不被判為錯誤。
4. RtnCode 型別（字串 vs 整數）判斷。
5. Webhook 冪等全組（IDM-01～10）。
6. 退款決策表全組合（REF-01）。
7. 對帳四類比對結果（REC-05～07）。
8. 環境切換開關：stage↔prod URL 與金鑰成組切換、無殘留混用。

### 2.3 回歸案例庫的維護規則

- **事故驅動擴充**：每次生產事故或綠界行為變化，先寫「再現案例」（失敗中）→ 修復 → 案例轉綠並永久保留。
- **官方文件驅動更新**：綠界「更新歷程」頁為監看對象；參數/端點異動時，先更新 `02-api-capability-matrix.md`，再同步對應案例。
- **案例與文件互相引用**：每個案例標注依據的藍圖章節；藍圖章節變更時可反查受影響案例。
- 淘汰：僅當官方明確移除某能力時封存對應案例，並在 `00-scope-and-sources.md` 註記。
