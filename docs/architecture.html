系統架構總覽

單一 Next.js 16 App Router 應用部署於 Vercel，資料層為單一 Supabase 專案，金流／電子發票走綠界 ECPay。沒有獨立後端——所有伺服器邏輯都在 Server Components／Actions／Route Handlers 內。本頁為 architecture.md 的儀表板視圖。

更新 2026-07-20（對齊 M2 進行中） 規模 16 表 · 4 RPC · 3 Cron · migrations 0001–0021 權威 程式碼與 docs/ 為準

03

資料庫端交易化（4 RPC）

多段寫入包進 plpgsql function＝同一交易單元，任一段失敗整段 rollback，消滅「半套狀態」中間態。RPC 是機制，是「塞進同一函式」讓它原子。

create_order_with_items0010

建 orders＋order_item 原子化（T76，防孤兒訂單）。

transition_order_status0017

狀態 CAS UPDATE＋order_status_log INSERT 原子化（T110，防「狀態已變、log 缺漏」）。

refund_order0020

payment 翻 refunded＋訂單 CAS＋稽核 log 原子化（T47）。

repair_refunded_payment0021

補登記退款（Override 逃生口的半套）：補翻 payment＋稽核 log 原子化（T47）。

04

金流兜底縱深

四層防線，每層守衛條件明確、靜默失敗點皆接 Sentry。付款判定以背景 webhook 為權威。

① Webhook（即時・權威）ReturnURL
驗章＋金額核對 → CAS 翻 payment → 推進訂單 RPC → 開票 → 補通知 → 1|OK
② 成功頁主動查詢（即時）QueryTradeInfo
結果頁未確認時主動打綠界快速對帳，數秒內給明確結果（三態：成功／失敗／處理中）
③ 對帳三臂＋2 sweep（每日 02:00）reconcile cron
主臂 payment=pending・打綠界
漂移臂 paid∧pending_payment・信任財務事實
稽核臂 paid∧{cancelled,refunded}・只偵測
sweep 補寄信＋補開票
④ email-pending（最後安全網）極少數
連訂單查詢都暫取不到時：「款項確認中，將以 email 通知你」，不提供重試
＊ 詳見 order-state-machine.html（狀態機＋三臂時序） ＊ 詳見 system-flow-and-user-flow.md 附錄 B

架構 Gap 狀態

自初版以來絕大多數缺口已落地。仍開放的架構級風險如下（依風險排序）。

🔴環境不分離：dev／staging／prod 全打同一顆 Supabase production DB；Vercel env 共用唯一真正的架構級風險——寫入面持續擴大前應優先解T82 · T83
🔴DB 備份未設定上線紅線（CLAUDE.md §6）T34
🟠Email 送達性：FROM 仍 onboarding@resend.dev、無 SPF/DKIM/DMARC上線前必換自有網域T35 · T50
🟠黑貓宅配未串接：shipping_fee=0 佔位運費與物流單號現況人工T48
🟠電子發票仍 sandbox：正式帳號＋字軌未設定上線前切正式T122
🟡E2E 測試套件未落地：CI 僅 typecheck/lint/unit金流閉環回歸靠人工T106
