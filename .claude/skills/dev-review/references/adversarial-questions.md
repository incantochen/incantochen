# 通用對抗性問題集（第一遍審查用）

讀每一段程式時帶著這些問題。這份問題集的目的是「產生」發現——清單只能「比對」發現。任何一題答不出來或答案令人不安，就是一個待查疑點，記下位置與情境。

## 1. 併發與競態

- 這段程式兩個請求同時進來會怎樣？check-then-insert／read-modify-write 有沒有 DB 層約束兜底？
- 狀態更新有沒有前置狀態守衛？兩個來源（webhook 與使用者操作）同時改同一列會怎樣？
- 「查了再用」的資料，在查與用之間被別人改掉會怎樣？

## 2. 信任邊界

- 這條路徑誰打得到？未登入的人？持有別人識別碼的人？直接 curl 的人？
- 每個輸入參數的來源是什麼？哪些值被當成「可信」而沒有驗證？（id、token、email、金額、狀態、URL 參數）
- 憑一個識別碼（order_no／id／email）就能做到什麼？該識別碼的猜測成本多高？
- 使用者輸入流向哪裡？（HTML／SQL／Email／log／外部 API）每個 sink 有沒有對應的跳脫或參數化？

## 3. 錯誤路徑

- 每一個外部呼叫（DB、金流、Email、Redis）失敗時，系統停在什麼狀態？誰會知道？
- catch 之後發生什麼？吞掉、記 log、還是讓呼叫方知道？「回報成功但實際失敗」的組合存在嗎？
- 多步驟寫入中途失敗，留下的半成品狀態誰清理？
- 重試安全嗎？外部系統（ECPay、Resend）收到重複請求會怎樣？

## 4. 冪等與重放

- 同一個 webhook／請求送兩次會怎樣？三次呢？隔很久再送呢？
- 冪等鎖在哪一層？（DB unique？程式判斷？）程式判斷的冪等在併發下還成立嗎？
- 副作用（寄信、扣款、清購物車）在冪等重複時會不會重複執行？

## 5. 生命週期與時間

- 每個「暫時狀態」（pending、cart、token、session）有沒有出口？永遠卡住會怎樣？
- 使用者在流程中途放棄（關瀏覽器、ECPay 頁面按上一頁），留下什麼？回得來嗎？
- 資料過期／改名／下架後，引用它的舊資料（訂單、購物車、email）顯示什麼？
- serverless：HTTP 回應送出之後還有什麼工作沒做完？那些工作保證會執行嗎？

## 6. 格式與轉換

- 任何字串切割／拼接／編碼轉換：兩端的格式假設一致嗎？格式演進過嗎（長度、前綴、隨機後綴）？
- 數字：小數、負數、極大值、NaN 進來會怎樣？金額在哪些點被轉換？
- 時區、編碼（中文進 shell／URL）、大小寫正規化有沒有一致？

## 7. 不對稱

- 「寫入路徑」加了的防護，「讀取路徑」有嗎？反過來呢？
- A 處修了的 bug，同模式的 B 處修了嗎？（同一份格式有幾個解析點？）
- 設計文件／schema 宣稱的機制（約束、去重表、稽核），程式真的在用嗎？
